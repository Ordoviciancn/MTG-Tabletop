## 项目可行性与开发规划报告

**项目代号：** MTG Tabletop  
**项目目录：** `D:\MTG-Tabletop`  
**项目定位：** 供朋友之间娱乐使用的非官方双人在线手动牌桌  
**文档版本：** 0.1  
**日期：** 2026-06-23

## 1. 执行摘要

本项目可以完成，而且在明确限制范围后，工程难度处于可控水平。

项目不尝试实现完整的《万智牌》规则引擎，也不自动判断一张牌能否施放、费用是否足够、触发式异能是否遗漏或复杂互动如何裁定。它更接近一个为《万智牌》优化过的在线桌游模拟器：服务器维护牌桌状态和隐藏信息，玩家负责遵守规则并通过语音或文字沟通。

推荐首版做成网页应用。玩家打开网页，创建或加入房间，分别导入自己的牌表，然后进行洗牌、抽牌、移动、横置、放置指示物、改变生命值等操作。

在这个边界内：

- 技术可行性：高
- Codex 独立完成代码的可行性：高
- 单机原型难度：低
- 双人联网 MVP 难度：中
- 可稳定娱乐使用版本难度：中
- 完整自动规则版本难度：极高，不属于本项目范围

建议先用 1 个短迭代完成双浏览器联机原型，再逐步补充断线重连、操作日志、撤销和更好的牌桌交互。

## 2. 产品目标

### 2.1 核心目标

让两名认识的玩家通过浏览器完成一局手动《万智牌》对战，并具备以下体验：

1. 无须安装专门客户端。
2. 通过房间码或邀请链接加入对局。
3. 双方可以导入自己的套牌。
4. 手牌及牌库顺序等隐藏信息只对所属玩家可见。
5. 所有影响牌局的随机行为由服务器执行。
6. 牌桌操作及时同步到双方浏览器。
7. 断线后能够回到原来的对局。
8. 玩家能查看操作记录，并在双方同意时撤销误操作。

### 2.2 非目标

首版明确不实现以下内容：

- 完整规则合法性判断
- 自动支付法术力
- 自动结算堆叠
- 自动处理触发式异能
- 自动生成衍生物和复制品
- 自动执行状态动作
- 自动处理规则层、替代效应及持续性效应
- 自动判断胜负
- 官方裁判级裁定
- 天梯、排名、匹配系统
- 商城、开包、收藏和交易
- AI 对手
- 大型公开锦标赛
- 防止熟人恶意作弊之外的专业反作弊体系

这些非目标必须保留在产品边界之外，否则工程量会迅速扩大。

## 3. 目标用户和使用场景

### 3.1 目标用户

- 已经了解《万智牌》规则的玩家
- 认识并愿意互相信任的朋友
- 想测试新牌表的玩家
- 无法线下见面但希望远程对战的玩家
- 愿意通过语音软件沟通优先权和裁定的玩家

### 3.2 不适合的用户

- 完全不懂规则的新手
- 希望系统自动指导所有操作的玩家
- 需要竞技级防作弊和裁判保障的公开比赛
- 希望获得与 MTGO 或 MTG Arena 相同自动化体验的玩家

### 3.3 典型使用场景

两名玩家通过语音通话连接。一名玩家创建房间并把邀请链接发给另一名玩家。双方导入牌表，系统检查基本数量和格式，但不判断全部赛制细节。双方确认准备后，服务器随机决定先后手，洗牌并发起对局。玩家在语音中宣告行动，通过牌桌完成抽牌、移动、横置、放指示物和生命值调整。

## 4. 完整用户流程

### 4.1 进入系统

首版可不要求注册账号。玩家输入临时昵称后进入首页。

可选入口：

- 创建房间
- 输入房间码加入
- 通过邀请链接直接加入

### 4.2 创建房间

房主设置：

- 房间名称
- 是否设置密码
- 起始生命值，默认 20
- 每局数量，首版只支持单局
- 是否允许观战，首版默认关闭
- 是否允许撤销，默认需要对方确认

系统生成：

- 6 至 8 位房间码
- 邀请链接
- 房间临时身份令牌

### 4.3 导入牌表

支持粘贴常见纯文本格式，例如：

```text
4 Lightning Bolt
4 Ragavan, Nimble Pilferer
2 Blood Moon

Sideboard
2 Engineered Explosives
```

系统执行：

- 解析数量和英文牌名
- 根据本地卡牌索引匹配卡牌
- 显示无法识别的行
- 将主牌和备牌分开
- 进行基本数量检查
- 缓存牌名、图片地址和必要元数据

首版不需要实现可视化组牌器。先实现可靠的牌表导入和保存到浏览器本地。

### 4.4 准备阶段

双方可以：

- 查看自己的牌表
- 替换牌表
- 标记准备
- 取消准备

两人都准备后才能开始。开始后服务器锁定当前牌表，生成牌张实例并洗牌。

### 4.5 起手和调度

流程建议：

1. 服务器为每名玩家抽 7 张牌。
2. 玩家选择保留或调度。
3. 调度时服务器将手牌洗回牌库并重新抽 7 张。
4. 玩家保留后，根据调度次数将相应数量的牌置于牌库底。
5. 双方确认后进入正式对局。

系统可以辅助伦敦调度的牌张数量，但不自动替玩家选择牌。

### 4.6 正式对局

牌桌包含：

- 双方生命值
- 双方牌库
- 双方手牌
- 双方战场
- 双方坟场
- 双方放逐区
- 公共堆叠区
- 回合和阶段指示
- 操作日志
- 骰子和硬币

基本操作：

- 抽牌
- 移动卡牌
- 横置或重置
- 翻面
- 调整卡牌朝向
- 调整生命值
- 添加或移除指示物
- 复制为临时衍生物
- 查看自己的牌库
- 搜索自己的牌库
- 展示指定卡牌
- 洗牌
- 将牌置于牌库顶或底
- 随机弃一张牌
- 将卡牌交给对方控制
- 将多张牌组成一叠
- 添加文字备注

### 4.7 结束对局

玩家可以：

- 认输
- 双方同意结束
- 重新开始并使用相同牌表
- 返回房间更换牌表

首版胜负结果只作展示，不进入排名。

## 5. 系统应强制执行的规则边界

虽然不实现完整规则引擎，系统仍应强制执行信息安全和状态一致性。

### 5.1 必须由系统强制的规则

- 玩家不能看见对手手牌内容。
- 玩家不能直接看见对手牌库内容及顺序。
- 洗牌必须由服务器完成。
- 随机弃牌、掷骰和硬币结果必须由服务器生成。
- 普通操作只能作用于自己有权限操作的牌张。
- 控制权转移后，新控制者可以操作该永久物。
- 所有操作必须经过服务器验证并广播。
- 服务器必须过滤发送给不同玩家的隐藏信息。
- 同一张牌必须拥有唯一实例 ID。
- 重连后必须恢复相同的公开与私有状态。

### 5.2 可由玩家自行遵守的规则

- 当前是否拥有优先权
- 是否满足施放时机
- 是否支付了正确费用
- 目标是否合法
- 是否应当触发某个异能
- 堆叠结算顺序
- 是否遗漏强制动作
- 牌张互动的最终裁定
- 是否已经达到规则规定的胜负条件

### 5.3 可选的轻量辅助

系统可以提供提示，但不阻止操作：

- 回合阶段按钮
- “请求响应”或优先权提示
- 本回合已经下过地的标记
- 当前主动玩家标记
- 操作前确认
- 常用快捷操作菜单

提示必须被描述为辅助信息，不能让用户误认为系统能够完整裁定规则。

### 6.3 权威服务器模型

客户端只提交意图，不直接修改最终牌局状态。

```text
客户端操作
    ↓
发送命令和预期状态版本号
    ↓
服务器验证身份、权限和版本
    ↓
服务器修改牌局状态
    ↓
写入事件日志
    ↓
按玩家权限过滤隐藏信息
    ↓
向双方广播新状态
```

例如，客户端不能发送“我的手牌现在是这五张”。它只能发送“把实例 A 从我的手牌移动到战场”。服务器确认实例 A 确实在该玩家手牌中，然后完成移动。

### 6.4 对局状态模型

建议的核心对象：

```text
Game
├─ gameId
├─ version
├─ players
├─ activePlayerId
├─ turnNumber
├─ phase
├─ sharedZones
├─ eventLog
└─ status

Player
├─ playerId
├─ nickname
├─ life
├─ library
├─ hand
├─ battlefield
├─ graveyard
├─ exile
└─ connectionState

CardInstance
├─ instanceId
├─ oracleCardId
├─ ownerId
├─ controllerId
├─ zone
├─ face
├─ tapped
├─ counters
├─ note
└─ position
```

牌张实例与卡牌定义必须分开。同名的四张牌共享卡牌定义，但每张牌有独立实例 ID、所在区域和状态。

### 6.5 隐藏信息模型

这是首版最需要认真设计的部分。

服务器不应把完整状态直接发给双方，再依赖前端隐藏。否则用户打开浏览器开发者工具即可看见对手手牌。

服务器应分别生成：

- 玩家 A 视图
- 玩家 B 视图
- 观战者视图

例如，对手手牌只发送数量和背面实例，不发送牌名、图片、卡牌 ID。对手牌库只发送数量。

### 6.6 同步和冲突处理

每个对局状态带有递增版本号。客户端命令包含自己操作时看到的版本号。

如果两名玩家几乎同时操作：

- 服务器按收到顺序执行。
- 与当前版本冲突的命令被拒绝。
- 客户端拉取最新状态后重试或提示用户。

在熟人娱乐环境中，不需要复杂分布式协同算法；单个权威服务器已经足够。

### 6.7 撤销机制

不建议直接允许任意回滚，因为撤销可能泄漏隐藏信息。

例如，玩家抽到一张牌后看到内容，再撤销抽牌，已经获得了额外信息。

推荐方案：

- 普通公开移动可以立即请求撤销。
- 涉及抽牌、查看牌库、随机结果等隐藏信息的操作标记为敏感。
- 敏感操作撤销必须得到对手确认。
- 撤销记录必须写入日志。
- 首版只支持撤销最近一个操作或最近一个操作组。

## 8. 主要风险与问题推导

### 8.1 范围膨胀

**问题：** 使用过程中很容易产生“顺便自动处理这个异能”的需求。

**推导：**

1. 自动处理一个异能需要理解牌张规则文本。
2. 规则文本会引用目标、费用、时机、区域和替代效应。
3. 一旦支持部分自动化，用户会默认其他相似互动也应当正确。
4. 系统开始承担裁定责任。
5. 项目会逐步变成规则引擎。

**应对：** 只实现与牌桌操作有关的通用动作，不为单张牌编写专属逻辑。

### 8.2 隐藏信息泄漏

**问题：** 如果前端收到完整状态，即使界面不显示，对手仍能读取数据。

**后果：** 牌局失去可信度。

**应对：** 服务器按身份过滤状态；为每种事件编写隐藏信息测试。

### 8.3 撤销导致额外信息

**问题：** 抽牌、检索、随机操作撤销后，玩家已经看到了本不该知道的信息。

**应对：** 敏感撤销要求对手确认，日志明确记录。

### 8.4 拖放交互复杂

**问题：** 卡牌需要拖放、缩放、横置、叠放、查看大图和适应不同屏幕。

**推导：** 牌桌界面虽然看似简单，但交互冲突会很多，例如拖动和点击、双击和缩放、移动和选择。

**应对：**

- 首版使用规则化区域布局，而不是完全自由画布。
- 卡牌在战场可自由移动，其他区域采用列表或扇形排列。
- 所有关键操作同时提供右键菜单，不能只依赖拖放。

### 8.5 网络断线和重复命令

**问题：** 玩家点击后网络中断，重连时不知道操作是否成功。

**应对：**

- 每个命令带唯一 commandId。
- 服务器保存已执行命令 ID。
- 重复命令返回原结果，不再次执行。
- 客户端重连后获取完整可见状态。

### 8.6 卡牌名称解析

**问题：** 牌表可能使用不同语言、旧名称、双面牌格式或拼写错误。

**应对：**

- 首版优先支持英文官方牌名。
- 对无法匹配的行提供明确错误。
- 后续增加中文名称映射和模糊搜索。
- 双面牌以统一 oracle ID 处理。

### 8.7 卡图加载和版权

**问题：** 卡牌图像、名称、符号和商标受相应权利保护。

**应对建议：**

- 项目保持非商业和小范围娱乐用途。
- 明确声明为非官方粉丝项目。
- 不声称获得官方授权。
- 不出售卡图、牌张或数字收藏。
- 卡图采用按需远程加载，不将全量高清卡图打包进项目。
- 提供关闭卡图、仅显示文字占位卡的能力。
- 若以后公开运营、收费或宣传，应先获得专业法律意见。

这部分是风险控制建议，不构成法律意见。

### 8.8 第三方数据源变化

**问题：** 卡牌数据接口可能限流、调整或暂时不可用。

**应对：**

- 使用批量数据生成本地卡牌索引。
- 不在每次页面操作时请求第三方接口。
- 定期手动或计划任务更新数据。
- 保留数据版本和更新时间。

### 8.9 玩家恶意操作

**问题：** 手动模拟器允许比真实规则更多的动作。

**应对：**

- 产品定位为熟人娱乐。
- 所有操作写入日志。
- 对敏感操作显示确认。
- 提供撤销请求和房主移除玩家功能。
- 不承诺竞技级反作弊。

### 8.10 移动端适配

**问题：** 手机上空间小，拖放容易误触。

**应对：** 首版优先支持桌面浏览器。移动端仅保证能够查看和进行基础操作，成熟后再设计专用布局。

### 8.11 状态模型过早复杂化

**问题：** 为未来可能出现的全部功能建立过度抽象的模型，会减慢首版开发。

**应对：** 首先支持最小区域、牌张实例和事件日志；通过明确类型和迁移脚本保持可扩展，而不是提前实现所有可能状态。

### 8.12 服务器费用

熟人娱乐流量很低，初期服务器资源需求不大。单个小型云服务器即可运行。

主要费用可能包括：

- 域名
- 小型云服务器
- 数据库或备份空间
- HTTPS 证书通常可免费获取

如果只在局域网或通过自建网络工具使用，可暂时不产生公网服务器费用。

## 9. 可行性推导

### 9.1 为什么这个版本可行

完整《万智牌》规则自动化困难，是因为系统必须理解并组合大量规则与卡牌互动。本项目移除了这项最昂贵的工作，只维护通用牌桌状态。

系统真正需要理解的是：

- 谁拥有哪张牌
- 谁当前控制哪张牌
- 卡牌位于哪个区域
- 哪些信息对谁可见
- 卡牌是否横置、翻面或带有指示物
- 哪个操作先发生

这些都能被清晰的数据模型和服务器命令表达，不需要理解牌张规则文本。因此，从“规则推理问题”降维成了“多人实时状态同步问题”，后者有成熟技术方案。

### 9.2 Codex 可独立承担的部分

Codex 可以完成：

- 产品与技术文档
- 项目脚手架
- 前端界面
- 后端房间服务
- WebSocket 协议
- 牌局状态模型
- 隐藏信息过滤
- 牌表解析
- 卡牌数据导入
- 自动化测试
- Docker 部署配置
- 使用说明
- 代码重构和故障修复

仍需要用户参与：

- 确认牌桌视觉偏好
- 实际玩几局并反馈操作是否顺手
- 决定部署位置和域名
- 提供或管理必要的服务器账号
- 对具体《万智牌》玩法体验进行验收
- 决定项目是否公开及承担相应合规决策

### 9.3 完成概率判断

在不加入完整自动规则的前提下：

- 完成本地可玩原型：很高
- 完成双人联网版本：高
- 完成朋友间稳定娱乐版本：高
- 完成公开运营级平台：中等，需要更多运维和安全工作
- 后续转为完整自动裁定平台：低，不建议作为自然延伸

成功的关键不是单个技术难点，而是持续守住产品边界。

## 12. 测试策略

### 12.1 单元测试

- 牌张实例在区域间移动
- 控制权与所有权分离
- 洗牌不改变牌张集合
- 抽牌减少牌库并增加手牌
- 牌表解析
- 状态版本递增
- 重复命令不重复执行

### 12.2 权限测试

- 玩家 A 看不到玩家 B 手牌内容
- 玩家 A 看不到玩家 B 牌库顺序
- 观战者看不到双方隐藏信息
- 非房间成员无法订阅对局状态
- 过期身份令牌无法执行命令

### 12.3 集成测试

- 创建房间到开始对局的完整流程
- 两个客户端实时同步
- 一方断线后重连
- 并发移动同一实例时只有一个成功
- 敏感操作撤销需要对方确认

### 12.4 端到端测试

使用两个独立浏览器上下文模拟两名玩家：

1. A 创建房间。
2. B 加入房间。
3. 双方导入牌表并准备。
4. 服务器开始对局。
5. A 抽牌，B 只能看到手牌数量变化。
6. A 将牌置入战场，B 能看到牌面。
7. B 将永久物横置，A 看到同步。
8. A 刷新页面并恢复对局。

### 12.5 人工试玩

自动化测试不能判断牌桌是否顺手。每个可玩里程碑后至少进行一局真实对战，记录：

- 最常用但操作步骤过多的动作
- 容易误触的区域
- 不清楚是否成功的操作
- 需要语音反复解释的界面状态
- 影响对局节奏的等待或动画

## 13. 安全和隐私

娱乐版本也应具备最低安全保障：

- 房间 ID 使用不可预测随机值。
- 房间码和内部 ID 分开。
- 每个玩家使用独立临时身份令牌。
- 所有输入在服务器验证。
- 限制单条牌表、昵称和聊天内容长度。
- 限制创建房间和发送命令频率。
- 生产部署使用 HTTPS/WSS。
- 日志不记录完整身份令牌。
- 对局过期后自动清理。
- 不收集不必要的个人信息。

首版不注册账号，可以显著降低隐私和账号安全负担。

## 14. 建议的决策

建议批准项目进入原型开发，但应冻结以下边界：

1. 只做双人对战。
2. 只做桌面浏览器优先。
3. 不做完整规则判断。
4. 不为单张牌编写专属规则逻辑。
5. 不做账号、排位和公开匹配。
6. 首版牌表以英文牌名导入。
7. 服务器负责隐藏信息、随机结果和状态同步。
8. 项目保持非商业、非官方娱乐定位。

首个开发目标应为：

> 两个浏览器通过房间码加入同一对局，分别导入牌表，由服务器洗牌和发牌，并能在不泄漏手牌信息的前提下同步卡牌移动、横置和生命值变化。

一旦这个纵向切片稳定，项目的主要技术风险就已经得到验证。

## 结论

这个娱乐版项目具备很高的实现可能性，Codex 可以承担绝大多数设计、编码、测试和部署工作。项目能否顺利完成，主要取决于是否坚持“手动桌游模拟器”的定位。

如果规则由玩家负责，系统只负责牌桌、权限、随机性和同步，那么项目是一个规模适中、边界清晰的实时 Web 应用。若逐步加入自动牌张规则，项目则会迅速跨入完全不同的复杂度等级。

因此，推荐从双浏览器联网纵向原型开始，并将隐藏信息正确性和断线恢复作为早期质量重点。
